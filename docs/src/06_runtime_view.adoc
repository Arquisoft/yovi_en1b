== Runtime View

Here are the main runtime scenarios that demonstrate how YOVI's components interact during gameplay.

The system consists of three components:

* *Frontend (FE)* — React + Vite, runs on port 80
* *Users Service (USERS)* — Node.js + Express + MongoDB, runs on port 3000
* *Gamey Logic (GAMEY)* — Rust game engine, runs on port 4000

---

=== Scenario 1: User Registration and Login

[plantuml,"auth-sequence",png]
----
@startuml
actor User
participant "Frontend" as FE
participant "Users Service" as USERS
participant "MongoDB" as DB

User -> FE: Fill in username and password
FE -> USERS: POST /createuser
USERS -> DB: Check username exists
DB --> USERS: Not found
USERS -> USERS: bcrypt hash password
USERS -> DB: Save user
DB --> USERS: User created
USERS --> FE: 201 { userId }
FE --> User: Redirect to login

User -> FE: Fill in username and password
FE -> USERS: POST /login
USERS -> DB: Find user by username
DB --> USERS: User found
USERS -> USERS: bcrypt compare password
USERS -> USERS: Sign JWT (24h)
USERS --> FE: 200 { token, userId, username }
FE --> User: Store token, redirect to game
@enduml
----

*Notable aspects:*

* Passwords are never stored in plain text — bcrypt with salt rounds 10
* JWT token expires after 24 hours and must be sent as `Authorization: Bearer <token>` on all protected requests
* Username uniqueness is enforced at DB level (unique index)

=== Scenario 2: Player vs Player Game

[plantuml,"pvp-sequence",png]
----
@startuml
actor Player
participant "Frontend" as FE
participant "Users Service" as USERS
participant "MongoDB" as DB

Player -> FE: Start new game vs player
FE -> USERS: POST /games\n{ board_size, game_type: PLAYER, name_of_enemy }
USERS -> USERS: Assign random starting turn (B or R)
USERS -> DB: Save game (status: IN_PROGRESS)
DB --> USERS: Game created
USERS --> FE: 201 { game }

loop Until game ends
    Player -> FE: Select cell (coordinates)
    FE -> USERS: POST /games/:id/move\n{ coordinates }
    USERS -> DB: Find game, get last yen_state
    DB --> USERS: Game state
    USERS -> USERS: Compute new yen_state\n(calls POST /compute on Gamey)
    USERS -> DB: Save move { player, coordinates, yen_state }
    DB --> USERS: Saved
    USERS --> FE: 201 { updated game }
    FE --> Player: Render new board state
end

Player -> FE: Game ends (win/loss/quit)
FE -> USERS: PUT /games/:id/finish\n{ result, yen_final_state, duration_seconds }
USERS -> DB: Update game status to FINISHED
USERS -> DB: Update user statistics (if WIN or LOSS)
DB --> USERS: Updated
USERS --> FE: 200 { finished game }
FE --> Player: Show result screen
@enduml
----

*Notable aspects:*

* Starting turn (B or R) is randomly assigned using `crypto.randomInt` — cryptographically secure
* Player field in each move is inferred from `current_turn` — frontend does not send it
* `yen_state` per move is computed by Gamey's `/compute` endpoint, not by the frontend or backend
* DRAW result (user quit) does not update statistics

=== Scenario 3: Player vs Bot Game

[plantuml,"pve-sequence",png]
----
@startuml
actor Player
participant "Frontend" as FE
participant "Users Service" as USERS
participant "MongoDB" as DB
participant "Gamey Logic" as GAMEY

Player -> FE: Start new game vs bot
FE -> USERS: POST /games\n{ board_size, game_type: BOT, strategy, difficulty_level }
USERS -> USERS: Assign random starting turn (B or R)
USERS -> DB: Save game
DB --> USERS: Game created
USERS --> FE: 201 { game }

loop Until game ends
    Player -> FE: Select cell (coordinates)
    FE -> USERS: POST /games/:id/move { coordinates }
    USERS -> DB: Get last yen_state
    USERS -> GAMEY: POST /compute\n{ yen_state_prev, coordinates }
    GAMEY --> USERS: { yen_state }
    USERS -> DB: Save player move
    USERS --> FE: 201 { updated game }
    FE --> Player: Render board

    FE -> USERS: GET /games/:id/play
    USERS -> DB: Get last yen_state
    USERS -> GAMEY: POST /play\n{ yen_state, strategy, difficulty_level, board_size }
    GAMEY --> USERS: { coordinates, yen_state }
    USERS -> DB: Save bot move automatically
    USERS --> FE: 201 { coordinates, yen_state }
    FE --> Player: Render bot move
end

Player -> FE: Game ends
FE -> USERS: PUT /games/:id/finish\n{ result, yen_final_state, duration_seconds }
USERS -> DB: Update game, update stats
USERS --> FE: 200 { finished game }
FE --> Player: Show result screen
@enduml
----

*Notable aspects:*

* The backend acts as a proxy between the frontend and Gamey — the frontend never calls Gamey directly
* `POST /compute` — Gamey receives `{ yen_state_prev, coordinates }` and returns the new `{ yen_state }` after applying the move
* `POST /play` — Gamey receives `{ yen_state, strategy, difficulty_level, board_size }` and returns the bot's `{ coordinates, yen_state }`
* Bot move is saved automatically in the DB by the Users Service — frontend does not need to call `POST /games/:id/move` for the bot
* If it is the first move, `yen_state_prev` / `yen_state` is sent as `null` and Gamey generates the initial board state from `board_size`

=== Scenario 4: Game Replay

[plantuml,"replay-sequence",png]
----
@startuml
actor Player
participant "Frontend" as FE
participant "Users Service" as USERS
participant "MongoDB" as DB

Player -> FE: View game history
FE -> USERS: GET /users/:id/history
USERS -> DB: Find games by player_id (no moves)
DB --> USERS: List of games
USERS --> FE: 200 [ games ]
FE --> Player: Show game list

Player -> FE: Select a game to replay
FE -> USERS: GET /games/:id/moves
USERS -> DB: Find game, return moves sorted by move_number
DB --> USERS: Moves array
USERS --> FE: 200 [ moves ]
FE --> Player: Replay move by move using yen_state per move
@enduml
----

*Notable aspects:*

* Each move stores its resulting `yen_state`, so the frontend can reconstruct the board at any point in time
* History endpoint excludes the moves array for a lighter response — full moves are fetched separately via `GET /games/:id/moves`
